# Provably-fair roll generation

This document specifies how KaniCasino generates and lets you verify rolls for
case opening, upgrade, slots, and case battles. The live games (crash, coin flip)
are out of scope; a shared live round can't use a per-user client seed.

> Status: implemented and tested across case opening, upgrade, slots, and case
> battles. Live crash / coin flip remain out of scope.

## The primitives

Implemented in [`backend/utils/provablyFair.js`](../backend/utils/provablyFair.js).

- **serverSeed**: 32 random bytes (hex), generated by a CSPRNG, kept **secret**.
- **serverSeedHash**: `sha256(serverSeed)`, published **before** any roll (the commitment).
- **clientSeed**: a string you control (defaults to a random hex value).
- **nonce**: an integer that increments by one for every roll under a seed.
- **cursor**: an extra index for games that need several draws from one nonce (slots).

A roll's uniform value is:

```
digest = HMAC_SHA256(key = serverSeed, message = `${clientSeed}:${nonce}:${cursor}`)
float  = digest[0]/256 + digest[1]/256^2 + digest[2]/256^3 + digest[3]/256^4   // in [0, 1)
roll   = floor(float * TOTAL) + 1                                              // in [1, TOTAL]
```

`TOTAL = 100_000`. The 4-byte fold is the widely-used "stake" construction, so any
third-party tool reproduces the same number.

### Reproduce a roll (any language)

```js
const crypto = require("crypto");
function rollFloat(serverSeed, clientSeed, nonce, cursor = 0) {
  const d = crypto.createHmac("sha256", serverSeed)
    .update(`${clientSeed}:${nonce}:${cursor}`).digest();
  return d[0]/256 + d[1]/256**2 + d[2]/256**3 + d[3]/256**4;
}
const roll = Math.floor(rollFloat(serverSeed, clientSeed, nonce) * 100_000) + 1;
```

To confirm the server didn't change the seed after the fact, check
`sha256(revealedServerSeed) === serverSeedHash` (the hash you were shown up front).

## Seed lifecycle (Phase 1)

- You always have your current `serverSeedHash`, `clientSeed`, and `nonce`, but
  **not** the current `serverSeed` (revealing it early would let you predict rolls).
- You can change your `clientSeed` at any time.
- On **rotate**, the old `serverSeed` is revealed (so you can verify every roll made
  under it), a new `serverSeed` is committed (new hash shown), and `nonce` resets to 0.
- You can therefore always verify **past** (rotated) seeds; your current seed is
  commit-only until you rotate it.
- A seed also **auto-rotates after 1000 rolls** (nonces), revealing it, so no seed
  ever governs more than 1000 outcomes before it becomes verifiable.

## Case opening: number → item (Phase 2)

A single roll picks an item via **ranges**: each item in a case owns a contiguous
range over `[1, TOTAL]`, sized to its probability. The item whose range contains the
roll is your drop.

An item's probability is `rarityChance / itemsOfThatRarityInTheCase`, renormalized
over the rarities the case actually contains (identical to today's odds when the case
has at least one item of every rarity).

Because the case's contents, order, rarities, and values are editable, the roll number
alone does **not** prove which item you got. So each roll's audit record snapshots the
**exact range table** it was resolved against, and each case publishes a versioned
config hash. Verification uses the snapshot, never the live case, so a later edit can't
rewrite history. (Worked example: ranges `A:1-10000`, `B:10001-55000`, `C:55001-100000`;
a roll of `66945` lands in `C`.)

## Upgrade / slots

Same primitives. Upgrade compares the roll float to the success rate; slots draw nine
symbols with `cursor = 0..8`. Each roll snapshots the effective threshold / paytable and
the algorithm version, so the outcome, not just the number, is reproducible.

## Case battles

A battle can't use one player's personal seed for a shared multi-player outcome, so it
commits its **own** server seed at start (hash exposed while it runs) and locks each
slot's client seed, a human's own client seed, or a deterministic `bot:{slot}` for
bots. Each slot's item for case `c` is `roll(battleServerSeed, clientSeed, nonce=c,
cursor=slot)` mapped through that case's range table. A tie is broken deterministically
from the battle seed (no stray randomness). When the battle finishes, the server seed is
revealed, so anyone can reproduce every player's items.

## Scope

Case opening, upgrade, slots, and case battles use this scheme. Crash and coin flip do
not (a shared live round needs a different, per-round commit-reveal construction).
